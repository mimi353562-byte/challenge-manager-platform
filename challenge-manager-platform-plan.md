# 챌린지 매니저 플랫폼 기획서

## 1. 문서 개요

- 문서명: 챌린지 매니저 플랫폼 서비스 기획서
- 작성일: 2026-06-11
- 목적: 블로그, 유튜브, 틱톡 기반의 챌린지를 생성, 운영, 검증, 정산, 랭킹화할 수 있는 통합 플랫폼의 서비스 방향과 핵심 기능을 정의한다.
- 현재 범위: 개발 착수 전 기획 단계

## 2. 서비스 한 줄 정의

참가비 기반 챌린지를 운영하고, 콘텐츠 인증과 성과 검증을 통해 성공 참가자에게 상금을 자동 분배하는 챌린지 운영 플랫폼

## 3. 서비스 비전

개인 창작자, 커뮤니티 운영자, 브랜드, 교육사업자가 직접 챌린지를 개설하고 운영할 수 있도록 하며, 참가자는 자신의 콘텐츠 활동을 통해 동기부여, 성취, 보상, 랭킹 경쟁을 동시에 경험할 수 있게 한다.

## 4. 핵심 문제 정의

기존 SNS 챌린지는 다음과 같은 운영 문제가 있다.

- 참가 신청과 참가비 수납이 수기 또는 외부 폼에 의존한다.
- 인증 기준이 불명확해 운영자 검수가 과도하게 발생한다.
- 블로그, 유튜브, 틱톡 등 플랫폼별 성과를 한 곳에서 관리하기 어렵다.
- 성공자 보상 정산이 수작업이라 누락과 분쟁 가능성이 높다.
- 우수 참가자 누적 기록과 시즌별 랭킹 운영이 어렵다.

## 5. 목표 사용자

### 5.1 운영자

- 챌린지를 기획하고 참가자를 모집하는 개인 운영자
- 콘텐츠 마케팅용 미션을 운영하는 브랜드
- 학습, 습관형 챌린지를 운영하는 코치 및 커뮤니티 리더

### 5.2 참가자

- 블로그 포스팅 챌린지 참가자
- 유튜브 업로드 챌린지 참가자
- 틱톡 숏폼 챌린지 참가자
- 다중 채널 성과를 누적 관리하고 싶은 크리에이터

## 6. 서비스 목표

### 6.1 비즈니스 목표

- 챌린지 개설부터 정산까지 한 플랫폼에서 완료
- 반복 운영이 가능한 시즌형 챌린지 구조 확보
- 참가비, 운영 수수료, 프리미엄 기능 기반 수익 모델 구축

### 6.2 사용자 목표

- 참가자는 쉬운 인증과 공정한 보상 체계를 원한다.
- 운영자는 검수 부담 감소와 자동 정산을 원한다.
- 전체 사용자는 신뢰할 수 있는 랭킹과 성과 이력을 원한다.

## 7. 핵심 가치 제안

- 통합 관리: 블로그, 유튜브, 틱톡 챌린지를 한 곳에서 운영
- 보상 설계: 참가비를 기반으로 성공자 상금 자동 분배
- 신뢰성: 인증 규칙, 검수 프로세스, 랭킹 산정 기준 표준화
- 확장성: 시즌제, 팀전, 스폰서형 챌린지까지 확장 가능

## 8. 핵심 서비스 구조

플랫폼은 크게 5개 영역으로 구성한다.

1. 챌린지 개설 및 모집

3. 인증 제출 및 검수

## 12. 챌린지 성공 및 상금 정책 제안

정산 분쟁을 줄이기 위해 정책을 명확히 정의해야 한다.

### 12.1 성공 기준 예시

- 블로그 챌린지: 기간 내 최소 10건 포스팅 + 필수 키워드 포함 + 링크 제출
- 유튜브 챌린지: 기간 내 최소 4개 업로드 + 공개 상태 유지 + 지정 해시태그 포함
- 틱톡 챌린지: 기간 내 최소 10개 업로드 + 지정 사운드 또는 해시태그 포함

### 12.2 상금 배분 예시

- 균등형
  - 참가자 100명, 참가비 1만원
  - 총 수입 100만원
  - 수수료 차감 후 80만원
  - 성공자 20명
  - 1인당 4만원 지급
- 차등형
  - 1위 30%
  - 2위 20%
  - 3위 10%
  - 나머지 성공자 균등 배분

### 12.3 주의할 정책 포인트

- 실패자의 참가비 미환급 여부 명시
- 중도 탈락자 처리 기준 명시
- 운영자 임의 지급 제한
- 신고 및 이의제기 기간 운영

## 17-1. 보안 관점에서 반드시 보완할 사항

현재 문서는 기능 중심으로는 충분한 출발점이지만, 실제 서비스 운영을 위해서는 아래 보안 항목이 필수로 추가되어야 한다.

### 17-1.1 계정 및 권한 보안

- 운영자, 일반 참가자, 플랫폼 관리자 권한을 분리하는 RBAC 구조 적용
- 운영자 계정 대상 2단계 인증 적용
- 로그인 이상 징후 탐지
  - 다중 지역 로그인
  - 짧은 시간 내 반복 로그인 실패
  - 관리자 권한 급상승 시도
- 중요 행위 재인증
  - 상금 지급 승인
  - 환불 승인
  - 챌린지 종료 처리

## 17-3. 외부 플랫폼 연동 관점의 현실적 제약

자동 검수와 랭킹 고도화를 위해서는 외부 플랫폼 연동을 고려해야 하지만, 실제로는 API 권한과 쿼터 제약이 있으므로 초기 기획에 이를 반영해야 한다.

### 17-3.1 유튜브 연동 제약

- 영상 ID 기반 메타데이터 조회는 API로 가능
- API 사용량 쿼터 관리 필요
- 대규모 조회 기능은 기본 쿼터를 빠르게 소모할 수 있음
- 쿼터 확장이 필요하면 별도 심사 및 정책 준수 검토가 필요함

### 17-3.2 틱톡 연동 제약

- 사용자 프로필 및 영상 조회는 사용자 권한 동의와 스코프 승인이 필요할 수 있음
- 모든 데이터 자동 수집을 전제로 기획하면 구현 리스크가 커짐
- 초기에는 링크 제출 + 수동 검수 중심 설계가 현실적임

### 17-3.3 블로그 연동 제약

- 블로그 플랫폼별 구조가 달라 자동 검수 규칙 표준화가 어려움
- 일부 플랫폼은 본문 구조 파싱 안정성이 낮을 수 있음
- 초기에는 URL, 캡처 이미지, 필수 키워드 수동 확인 병행이 적절함

## 17-4. 운영 안정성을 위한 추가 기능 제안

- 정산 보류금 또는 예비비 정책
  - 환불 분쟁, 지급 실패, 세액 조정 대응 목적

  - 신규 운영자
  - 검증 운영자
  - 제재 이력 운영자
- 참가자 신뢰 점수
  - 허위 인증 이력 반영
  - 반복 반려율 반영
- 고객센터/문의 티켓 기능
- 공지사항 열람 확인 기능
- 신고 접수 SLA 및 처리 상태 추적

## 21. 웹 확인 기반 최신 반영 메모

아래 항목은 2026-06-11 기준 공개 문서를 확인해 기획서 보완에 반영한 내용이다.

- 국세청 안내 기준으로 기타소득이 125,000원을 초과하는 경우 지급액의 8.8% 원천징수 안내가 있으므로, 상금 지급 정책과 정산 로직에 세무 처리 분기가 필요하다.
- YouTube Data API는 기본 쿼터 체계를 운영하며, 기본 할당 및 추가 쿼터 심사 정책이 있으므로 대량 자동 검수 구조는 초기부터 쿼터 설계를 포함해야 한다.
- YouTube `videos.list`는 1회 호출당 1쿼터이지만 대규모 영상 검수 시 누적 사용량이 커질 수 있으므로, 챌린지 단위 배치 검수 설계가 필요하다.
- TikTok Display API는 사용자 권한 승인과 `user.info.basic`, `video.list` 등의 스코프가 필요하므로, 초기 서비스는 링크 제출 기반 검수와 선택적 계정 연동을 병행하는 설계가 현실적이다.
- 개인정보 관련 정책은 최소 수집, 목적 명확화, 보유기간 및 파기 정책을 별도 운영 문서로 구체화해야 한다.
