- 작성일: 2026-06-11

- 범위: 1차 출시 버전(MVP) 기준

## 2. 디자인 목표

이 플랫폼의 디자인은 단순히 보기 좋은 UI가 아니라, 아래 3가지를 동시에 해결해야 한다.

1. 참가자가 지금 무엇을 해야 하는지 즉시 이해할 수 있어야 한다.
2. 운영자가 많은 인증 제출과 정산 작업을 빠르게 처리할 수 있어야 한다.
3. 결제, 검수, 상금, 랭킹이 얽힌 서비스 특성상 신뢰감과 투명성이 강하게 느껴져야 한다.

## 3. 제품 디자인 핵심 방향

### 3.1 참가자 경험 방향

- 도전 욕구를 자극하는 구조
- 현재 진행 상태가 한눈에 보이는 구조
- 인증 제출이 최대한 빠른 구조
- 성공 가능성과 보상 기대치를 쉽게 이해하는 구조

### 3.2 운영자 경험 방향

- 검수, 공지, 정산을 빠르게 처리하는 구조
- 표와 카드가 적절히 혼합된 관리형 인터페이스
- 실수 방지를 위한 단계적 확인 구조
- 반복 운영에 최적화된 템플릿형 구조

### 3.3 서비스 인상

브랜드 톤은 "가벼운 커뮤니티 앱"보다 아래에 가까워야 한다.

- 신뢰 가능한 운영 도구
- 성과가 보이는 생산성 플랫폼
- 챌린지를 게임처럼 즐기게 하되, 정산 화면에서는 금융 서비스처럼 명확한 경험

## 4. 핵심 UX 원칙

### 4.1 한 화면 한 목적 원칙

- 챌린지 탐색 화면은 탐색에 집중
- 인증 제출 화면은 제출에 집중
- 검수 화면은 판정에 집중
- 정산 화면은 계산과 확정에 집중

### 4.2 상태 우선 원칙

이 플랫폼에서 가장 중요한 UX는 "상태가 분명하게 보이는 것"이다.

- 참가 상태
- 제출 상태
- 검수 상태
- 성공/실패 상태
- 정산 상태

모든 주요 화면은 정보보다 상태가 먼저 보이도록 설계한다.

### 4.3 불확실성 최소화 원칙

참가자는 아래 질문에 항상 즉시 답을 얻어야 한다.

- 내가 지금 참가 완료된 상태인가?
- 이번 회차 인증을 제출했는가?
- 승인되었는가, 반려되었는가?
- 앞으로 몇 번 더 제출해야 성공인가?
- 예상 상금은 얼마인가?

운영자는 아래 질문에 즉시 답을 얻어야 한다.

- 오늘 검수해야 할 건이 몇 개인가?
- 반려가 많은 회차는 어디인가?
- 정산 확정 전 누락된 검수는 없는가?
- 현재 지급 보류 건은 무엇인가?

### 4.4 실수 방지 원칙

- 결제, 환불, 정산 확정, 지급 승인처럼 되돌리기 어려운 행위는 모두 확인 단계 필요
- 운영자 액션 버튼은 시각적 우선순위를 명확히 구분
- 참가자 제출 시 필수 항목 누락을 사전에 차단

### 4.5 모바일 우선 제출 원칙

참가자의 주요 액션은 모바일에서 발생할 가능성이 높다.

- 탐색
- 결제
- 링크 복사/붙여넣기
- 이미지 첨부
- 제출 확인

따라서 참가자 화면은 모바일 우선으로 설계하고, 운영자 화면은 데스크톱 우선으로 설계한다.

## 9. 시각 디자인 방향

### 9.1 브랜드 성격

- 활기 있음
- 경쟁적임
- 명확함
- 신뢰 가능함

### 9.2 권장 비주얼 톤

참가자 영역:

- 에너지가 느껴지는 강조색 사용
- 챌린지, 랭킹, 달성률에서 게임적 긴장감 표현

운영자/관리자 영역:

- 보다 절제된 색 사용
- 데이터 판독성과 액션 명확성을 우선

### 9.3 컬러 전략

권장 방향:

- 기본색: 블루 또는 딥블루 계열
  - 신뢰, 구조, 플랫폼 인상
- 강조색: 오렌지 또는 레드오렌지 계열
  - 챌린지, 제출, 랭킹, CTA 강조
- 성공색: 그린
- 경고색: 앰버
- 반려/위험색: 레드

컬러 원칙:

- 참가자 액션 CTA와 운영자 승인 CTA는 같은 색을 쓰지 않는다.
- 정산/결제 관련 정보에는 과도한 장식보다 명확한 구분을 우선한다.

### 9.4 타이포그래피 방향

- 정보 밀도와 가독성을 우선
- 참가자용은 명확하고 직관적인 헤드라인
- 운영자용은 숫자/상태/표 정보 판독이 쉬운 서체

권장 구성:

- 헤드라인: 강한 굵기
- 본문: 중간 굵기
- 보조 정보: 작은 크기지만 대비 충분히 확보
- 숫자/금액: 고정폭 숫자 또는 숫자 가독성이 높은 스타일 사용

## 12. 상태 표현 설계

### 12.1 챌린지 상태 표현

- 초안
- 모집중
- 모집마감
- 진행중
- 검수중
- 결과확정대기
- 정산중
- 종료
- 중단

표현 원칙:

- 단순 텍스트가 아니라 배지 형태로 표현
- 중단, 정산중, 결과확정대기 상태는 시각적으로 강하게 구분

### 12.2 제출 상태 표현

- 제출완료
- 검수중
- 승인
- 반려
- 재제출요청

표현 원칙:

- 승인: 안정적 성공색
- 반려: 주의색이 아닌 명확한 위험색
- 재제출요청: 경고색

### 12.3 정산 상태 표현

- 계산완료
- 운영자확정
- 관리자승인
- 지급중
- 지급완료
- 지급실패
- 보류

표현 원칙:

- 결제/정산 영역은 상태 변화가 사용자의 신뢰와 직결되므로 타임라인 + 배지 조합 추천

## 13. 핵심 화면 기획

### 13.1 메인 홈

목표:

- 신규 사용자는 서비스 구조를 이해
- 기존 사용자는 바로 액션 진입

구성:

- 상단 히어로
- 추천 챌린지
- 진행 중 챌린지
- 인기 랭킹 미리보기
- 운영자용 개설 유도 섹션

### 13.2 챌린지 탐색 화면

목표:

- 참가자가 자신에게 맞는 챌린지를 빠르게 찾게 함

구성:

- 검색
- 플랫폼 필터
- 참가비 필터
- 난이도 필터
- 모집중/진행중 필터
- 챌린지 카드 리스트

### 13.3 챌린지 상세 화면

가장 중요한 참가 전환 화면이다.

반드시 보여야 할 것:

- 챌린지 핵심 요약
- 성공 조건
- 참가비
- 상금 방식
- 예상 일정
- 운영자 정보
- 환불/반려/이의제기 규칙
- 참가 CTA

### 13.4 결제 화면

목표:

- 결제 이탈 최소화
- 분쟁 방지

구성:

- 결제 요약
- 정책 동의
- 환불 규칙 확인
- 최종 결제 CTA

### 13.5 내 챌린지 화면

목표:

- 참가자가 현재 해야 할 행동을 즉시 알게 함

구성:

- 현재 참여 챌린지
- 회차별 제출 현황
- 남은 필요 제출 수
- 최근 반려/재제출 요청
- 결과 및 랭킹 바로가기

### 13.6 인증 제출 화면

MVP에서 가장 자주 쓰이는 화면이다.

구성:

- 회차 정보
- 제출 마감 시간
- 링크 입력 필드
- 이미지 첨부
- 메모 입력
- 제출 가이드
- 제출 버튼

중요 UX:

- 모바일에서 링크 붙여넣기와 이미지 첨부가 쉬워야 함
- 제출 성공 후 즉시 상태 피드백 제공

### 13.7 랭킹 화면

목표:

- 경쟁 심리 유도
- 공정한 기준 제공

구성:

- 랭킹 기준 설명
- 현재 순위
- 상위 랭커 리스트
- 내 순위 하이라이트

### 13.8 운영자 대시보드

목표:

- 운영자가 오늘 해야 할 일을 바로 이해

구성:

- 진행 중 챌린지 수
- 검수 대기 건수
- 반려율
- 모집 현황
- 정산 예정 건
- 이의제기 건수

### 13.9 검수 큐 화면

운영자 UX에서 가장 중요하다.

구성:

- 상태/회차/챌린지 필터
- 제출 리스트
- 상세 미리보기
- 승인/반려/재제출요청 버튼
- 반려 사유 입력

### 13.10 정산 관리 화면

정산 화면은 일반 대시보드와 다르게 "확인" 중심 구조여야 한다.

구성:

- 총 결제금액

## 15. 접근성 및 사용성 원칙

- 색만으로 상태를 구분하지 않는다.
- 버튼, 링크, 경고 메시지는 대비를 충분히 확보한다.
- 정산/결제 정보는 숫자 판독이 쉬워야 한다.
- 오류 메시지는 무엇이 잘못됐는지 구체적으로 알려준다.
- 제출 마감, 정산 확정 등 시간 정보는 절대 시각 기준으로 표기한다.

## 19. 최종 방향 요약

이 플랫폼의 디자인은 "챌린지를 재미있게 보이게 하는 것"만으로는 부족하다.  
참가자에게는 동기부여와 명확한 진행 상태를, 운영자에게는 빠른 검수와 정산 신뢰성을, 관리자에게는 통제와 추적 가능성을 제공해야 한다.

따라서 디자인의 핵심은 아래 세 가지다.

1. 상태가 먼저 보이는 인터페이스
2. 참가자와 운영자의 사용 맥락이 명확히 구분된 인터페이스
3. 경쟁과 보상은 강하게, 정산과 운영은 명확하게 보이는 인터페이스
